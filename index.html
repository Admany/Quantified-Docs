Welcome to Quantified API

So you're building a Minecraft mod and your async code is doing that fun thing where it works 90% of the time, and the other 10% it just... detonates a nuke or hates other mods. Threads shooting each other, futures vanishing into thin air, and best of all tick threads crying LMAO

Quantified API is a runtime layer that handles scheduling, threading, GPU compute, caching, and observability. You write just the interface code and that's all. Quantified API figures out the rest: where and when to run it, and how ;]

How it actually works

The internals are interestin. Handles, routers, async managers.

But WHY does this exist

Simple: there is nothing like it. Mods love to bolt CompletableFuture.runAsync directly onto vanilla code paths expecting it'll work fine (it 99% of the times doesn't...). That scatters work across unnamed thread pools, competes with the server ticks, duplicates the same task across EVERY module, and eventually DETONATES when something blocks on the main thread waiting for a future that resolves slightly later than expected. And then the TPS graph goes brrrr downwards or the game just dies...

Quantified API gives you one central place for identity (per-mod handles so work is always attributed), stable keys (deduplication and cache locality), routing (CPU vs GPU with honest fallbacks), and observability (dashboard + telemetry so you can actually see what’s happening). You still write the math while Quantified API owns the spaggeti.

V2 will auto-register your mod on first real use by pulling the mod ID, display name, and version from loader metadata. That auto-detection path supports Fabric, Forge, and NeoForge loader metadata. Manual register(...) is now mostly an override or an escape hatch for weird bootstrap/shared-helper situations.

Pick the right tool for the job

What you needWhat to useWhy
One async result QuantifiedAPI.compute(name) Fluent builder. Supports GPU workloads with CPU fallback.
Many items in parallel parallel(...).items(...).map(...) Controlled multitasking, configurable failure policies.
Index-based loops parallel(...).range(a, b).map(...) Same engine, no intermediate list allocation.
Complex pipelines with deps QuantifiedAPI.graph(modId, name) DAG execution, context.result(handle) for wiring.
Cache expensive pure work cache("bucket").get(key, loader) TTL, size caps, optional disk persistence.
Fire-and-forget side effect QuantifiedAPI.runAsync(name, () -> ...) Thin wrapper over compute, no return value.
Prefer composing CompletableFuture with thenApply and exceptionally instead of calling .join() on the server thread. Blocking there shows up as lag. If you genuinely need to block in a safe window (world load, data fixer), comment it in code so the next person knows it’s intentional.

Let’s get you up and running

Three steps. Actually like two and a half xd. You’ll be submitting async jobs within 5 minutes (that is if you follow a long...).

Step 1 - Usually do nothing :DDD

In V2, Quantified API auto-registers your mod the first time you actually use compute, parallel, graph, or cache. It grabs your mod ID, display name, and version from the loader metadata automatically on Fabric, Forge, and NeoForge.

Java
// Normal V2 usage: just use the API. // QAPI auto-registers your mod from loader metadata. CompletableFuture<Digest> future = QuantifiedAPI .compute("chunkDigest") .submit(() -> expensiveHash(snapshot)); // Optional manual override if you genuinely need it: QuantifiedAPI.register("mymod"); QuantifiedAPI.register("mymod", "My Cool Mod", "1.0.0");
If you are in shared utility code, test scaffolding, or some cursed bootstrap edge-case where auto-detection cannot tell which mod owns the call, use the explicit modId overloads or manual register(...).

Step 2 - Submit some work

CPU compute is the most common case. You get back a CompletableFuture. Chain on it, don’t block on it from the tick thread.

Java - async compute
CompletableFuture<Digest> future = QuantifiedAPI .compute("chunkDigest") // task name (for metrics + keys) .key(dimId + "|" + cx + "|" + cz) // stable dedup key .foreground() // or .background() for low priority .submit(() -> expensiveHash(snapshot)); future.thenAccept(digest -> { // runs off the tick thread - safe! storeResult(digest); });

Step 3 - Parallel over a list (the WORST part)

Java - parallel map
CompletableFuture<List<VoxelMesh>> meshes = QuantifiedAPI .parallel("bakeMeshes") .maxParallelism(Math.min(8, Runtime.getRuntime().availableProcessors())) .items(partitions) .map(p -> bake(p)) .submit(); // Range-based (no list needed): QuantifiedAPI.mapRange("processRows", 0, height, i -> processRow(i));

Handy shorthands

  • QuantifiedAPI.forEach(name, collection, consumer) - parallel side effects, returns when all done.
  • QuantifiedAPI.mapOrdered(name, collection, mapper) - parallel but preserves list order.
  • QuantifiedAPI.mapRange(name, start, end, fn) - index-based parallel, no stream boilerplate.
  • QuantifiedAPI.runAsync(name, runnable) - fire and forget, no result.
If your mod separates bootstrap from execution contexts (common in multi-module setups), use the explicit modId overloads: compute(modId, task), parallel(modId, task), graph(modId, name). That bypasses auto-detection and avoids confusion when helper classes are shared between multiple mods.

Before you ship, check these

  • Every hot-path task has a stable key when the same task name is reused across different inputs.
  • GPU code paths have a CPU fallback that produces equivalent output.
  • Your parallel failure policy matches what you actually want (FAIL_FAST vs BEST_EFFORT).
  • You’ve opened developerDashboard: true at least once.

Migrating from the V1 API

V2 aims to double down on what actually matters - async, parallel multitasking, and GPU routing while being less messy. The API is now even cleaner, more explicit, and actually matches how the runtime thinks.

The big shift? V2 is now builder first. Instead of a hundred static helpers, you start with compute, parallel, or cache and build your intent from there.
Registration changed too: V1-style eager register(...) is no longer the normal path. In V2, Quantified API auto-registers from loader metadata on Fabric, Forge, and NeoForge the first time you actually use the API. Keep manual register(...) only for explicit overrides or ambiguous bootstrap/shared-helper contexts.

The TLDR

V1 was fragmented badly and prone to accidental behavior. V2 on the other hand demands explicit keys and shapes, giving you one obvious path for every job.

What’s actually different?

Builders are now THE law

V1 was full of ad-hoc helpers everywhere. V2 on the other hand, centers everything on fluent request objects. It keeps your code readable even when you’re piling on timeouts, fallbacks, and GPU preferences.

Identity is now everything

Stable keys aren’t an afterthought anymore as before. If your old code used random or hidden keys, switch to explicit key(...) values based on your inputs. It’s a massive performance win of V2.

Intent over implementation

Fully forget digging for Vulkan or OpenCL entry points. In V2, you just call compute() and then tag on what you want: preferGpu(), preferVulkan(), or cpuOnly(). You describe the goal, the API handles the scheduling, submission, processing and sends the results back to ya.

V1 to V2 mapping

V1 QuantifiedAPI.register(...) as required setup
V2 Usually nothing; auto-register on first API use

Manual registration is now mainly an override or escape hatch for ambiguous ownership.

V1 QuantifiedAPI.submit(...)
V2 QuantifiedAPI.compute(name)...submit(...)

Normal async work now goes through the compute builder.

V1 QuantifiedAPI.task(...)
V2 QuantifiedAPI.compute(...)

Task-builder style was folded into the fluent compute API.

V1 QuantifiedAPI.submitParallel(...)
V2 QuantifiedAPI.parallel(...)

Use items(...) or range(...) depending on shape.

V1 ParallelCompute.Builder as primary entry
V2 QuantifiedAPI.parallel(...)

Parallel is still there, but it is now a first-class public lane.

V1 QuantifiedAPI.getCached(...)
V2 QuantifiedAPI.cache(bucket).get(key, loader)

Cache buckets are explicit now.

V1 QuantifiedAPI.getCachedAsync(...)
V2 QuantifiedAPI.cache(bucket).getAsync(key, loader)

Async cache population lives on the cache builder.

V1 QuantifiedAPI.putCached(...)
V2 QuantifiedAPI.cache(bucket).put(key, value)

Same behavior, clearer location.

V1 QuantifiedAPI.vulkan(...)
V2 QuantifiedAPI.compute(...).preferVulkan()

Most code should express preference, not start from a backend namespace.

V1 QuantifiedAPI.hybrid(...) / hybridBuilder(...)
V2 compute(...).preferGpu().fallback(...)

Hybrid routing still exists conceptually, but it is now described as intent on the compute request.

V1 Implicit/random task identity
V2 key(...)

Use stable logical keys if you want dedupe, joining, or cache-friendly behavior.

Common call-site rewrites

These are the kinds of changes you will make most often when moving old mods onto V2.

1. Basic async task

Java
// V1 CompletableFuture<ChunkData> future = QuantifiedAPI.submit("chunkData", () -> buildChunkData(pos)); // V2 CompletableFuture<ChunkData> future = QuantifiedAPI.compute("chunkData") .key(pos.toLong()) .submit(() -> buildChunkData(pos));

2. Helper code outside the main mod class

Java
// V1 style helpers often depended on hidden mod context // and could break if called from the wrong place. // V2: use the explicit mod overload when you are in shared utilities. CompletableFuture<Digest> future = QuantifiedAPI.compute("mymod", "digestAssets") .key(resourcePackVersion) .background() .submit(() -> digestAssets());

3. Parallel multitasking

Java
// V1 CompletableFuture<List<Mesh>> meshes = QuantifiedAPI.submitParallel("meshBuild", chunks, this::buildMesh); // V2 CompletableFuture<List<Mesh>> meshes = QuantifiedAPI.parallel("meshBuild") .items(chunks) .map(this::buildMesh) .submit();

4. Index-based loops

Java
// V2 prefers range-based parallelism for index loops. CompletableFuture<List<Float>> samples = QuantifiedAPI.parallel("sampleNoise") .range(0, count) .map(i -> sampleNoise(seed, i)) .submit();

5. Cache access

Java
// V1 ChunkPlan plan = QuantifiedAPI.getCached(cacheKey, () -> buildPlan(snapshot)); // V2 ChunkPlan plan = QuantifiedAPI.cache("chunk-plans") .ttl(Duration.ofMinutes(15)) .maxEntries(10_000) .get(cacheKey, () -> buildPlan(snapshot));

6. GPU or hybrid workloads

Java
// V1 backend-first / hybrid-ish style // QuantifiedAPI.vulkan(...) // QuantifiedAPI.hybrid(...) // V2 intent-first style CompletableFuture<float[]> density = QuantifiedAPI.compute("densityPass") .key(kernelKey) .preferGpu() .preferVulkan() .fallback(() -> runCpuDensityPass(input)) .submit(() -> runPreferredDensityPass(input));

7. Graph creation

Java
// V2 graph API is still graph-centric, but should be treated as the // first-class way to express staged dependency work rather than just // another quirky helper. QuantifiedTaskGraph.Builder graph = QuantifiedAPI.graph("lightingPipeline"); var gather = graph.node("gather", () -> gatherInputs()); var solve = graph.node("solve", ctx -> solve(ctx.result(gather))).dependsOn(gather); CompletableFuture<LightState> result = graph.submit(solve);

Migration rules that save you pain

Always add stable keys when the work can repeat

If your V1 code relied on unnamed or hidden identity, V2 is the moment to fix that. Keys should be derived from actual logical inputs: chunk position, dimension, seed, resource version, player UUID, config digest, and so on.

Use explicit mod-id overloads in shared helpers

Utility classes, static helpers, or shared cross-mod code should not depend on thread-local registration context. Use the overloads that take modId explicitly so the request is always bound correctly.

Do not translate everything one-to-one blindly

Some V1 entry points existed because the old architecture was more fragmented. If you were using Vulkan-first or hybrid-first submission as the default style, don’t just mechanically rename it - rethink whether the call site should really be a normal compute(...) request with GPU preference instead.

Move cache policy onto the cache bucket, not random callers

V2 wants cache ownership to be obvious. If a bucket needs TTL, persistence, size caps, or manual invalidation, put that policy on the bucket builder rather than scattering it across unrelated call sites.

Recommended migration order

  1. Update registration and helper code to use the correct V2 entry points.
  2. Convert plain async submission to compute(...).
  3. Convert parallel multitasking to parallel(...).
  4. Move cache usage onto explicit cache(bucket) builders.
  5. Add stable keys everywhere repeated work is possible.
  6. Only after that, revisit GPU preference and graph call sites for better V2-native structure.
If a migration feels awkward, that is usually a sign the old call site was depending on V1 weirdness rather than expressing the actual shape of the work. In V2, try to ask: is this one async compute job, one parallel workload, one graph pipeline, or one cache-backed lookup?

Code examples from LC²H

Here are some code examples from LC²H to get ya moving. The public version of LC²H on GitHub is still on V1, but these snippets are pulled straight from the unpublished dev branch of mine :]

Source material for this page comes from LC²H systems like LostCityDagScheduler, ParallelWorkQueue, AsyncBuildingInfoPlanner, and LostCitiesCacheBridge.
Notice that these examples use the explicit modId overloads. That’s intentional. LC²H does a lot of work from helper classes, schedulers, and background systems, so the auto-detector tends to be wonky.
Example A - LC²H foreground compute for chunk-stage planning

This comes from LC²H’s spawn and building-info scheduling kinda stuff. The real job here is simple: throw expensive chunk planning off-thread, keep it foreground because the player is literally waiting on it, and don’t let repeated requests for the same chunk multiply into dumb duplicate work.

In V2 terms this should just be compute(...) with a real key and affinity. Not some weird ad-hoc task builder chain.

Java
private static CompletableFuture<Void> submitSpawnTask( IDimensionInfo provider, ChunkCoord coord, Runnable action ) { String taskKey = provider.getType().toString() + ":" + coord.chunkX() + ":" + coord.chunkZ(); String locality = provider.getType().toString() + ":chunk:" + coord.chunkX() + ":" + coord.chunkZ(); return QuantifiedAPI .compute(LC2H.MODID, "spawn-buildinginfo-batch") .key(taskKey) .affinity(locality) .foreground() .timeout(Duration.ofSeconds(15)) .submit(() -> { action.run(); return null; }); }
  • key(...) stops duplicate chunk requests from multiplying for no reason.
  • affinity(...) keeps the work local to the chunk region or dimension instead of tossing it around randomly.
  • foreground() is right here because this is player-visible work, not some background cleanup slop.
Example B - LC²H parallel slice execution for region or batch work

LC²H has a reusable parallel work queue for region or multichunk work. It caps parallelism on smaller batches, can stream partial results out, and has fallback logic when pressure gets dumb. In V2, the core shape of that work is still just parallel(...).

Java
CompletableFuture<List<MultiChunk>> future = QuantifiedAPI .parallel(LC2H.MODID, "lc2h-multichunk-batch") .key(dimensionId + ":batch:" + startChunkX + ":" + startChunkZ) .maxParallelism(Math.min(4, Runtime.getRuntime().availableProcessors())) .items(multiCoords) .map(coord -> kernel.planMultiChunk(inputFor(provider, coord, areaSize)).multiChunk()) .stream(chunk -> telemetry.recordPreparedChunk(chunk)) .submit();
  • This is basically the V2 version of LC²H’s old ParallelCompute.Builder-first queue.
  • stream(...) is nice when you want progress reporting or side bookkeeping while the batch is still going.
  • If the work is index-driven instead of list-driven, switch it to range(start, end).
Example C - LC²H DAG scheduling for multichunk worldgen

LC²H already has a real DAG scheduler for kernel-driven multichunk planning. This is one of the stronger examples in the codebase because it uses locality, wavefront execution, node-level timeouts, and a gather stage that actually reduces dependency results instead of pretending to be a graph.

Java
QuantifiedTaskGraph.Builder graph = QuantifiedAPI .graph(LC2H.MODID, "lc2h-kernel-multichunk-batch") .key(dimensionId + ":batch:" + startChunkX + ":" + startChunkZ) .locality(dimensionId + ":batch:" + startChunkX + ":" + startChunkZ) .wavefront(); ArrayList<QuantifiedTaskGraph.NodeHandle<LostCityMultiChunkPlan>> handles = new ArrayList<>(); for (int i = 0; i < multiCoords.size(); i++) { ChunkCoord coord = multiCoords.get(i); LostCityKernelInput input = inputFor(provider, coord, areaSize); handles.add(graph .node("plan-multichunk-" + i, ctx -> kernel.planMultiChunk(input)) .critical() .threadSafe(true) .timeout(Duration.ofSeconds(30)) .batchKey("lc2h:kernel:multichunk-batch:" + dimensionId) .localityKey(input.localityKey())); } QuantifiedTaskGraph.NodeHandle<List<MultiChunk>> gather = graph .node("gather", ctx -> { List<MultiChunk> out = new ArrayList<>(handles.size()); for (QuantifiedTaskGraph.NodeHandle<LostCityMultiChunkPlan> handle : handles) { out.add(ctx.result(handle).multiChunk()); } return out; }) .dependsOn(handles.toArray(new QuantifiedTaskGraph.NodeHandle<?>[0])) .critical() .threadSafe(true); CompletableFuture<List<MultiChunk>> future = graph.submit(gather);
  • Use a graph when the dependency structure is actually real, not just because the method got long and ugly.
  • wavefront() fits nicely when a bunch of sibling nodes can run together before a final gather or reduce step.
  • Node-level batchKey and localityKey let the runtime keep better scheduling behavior than some flat future chain nonsense.
Example D - LC²H cache bridge for reusable planning artifacts

LC²H has a cache bridge for expensive Lost Cities derived state. In V2 this gets way cleaner: use an actual named cache bucket, make persistence explicit, and let the API deal with async cache population instead of hiding weird cache logic all over the place.

Java
public static CompletableFuture<BuildingInfo> getOrScheduleBuildingInfo( IDimensionInfo provider, ChunkCoord coord ) { String key = provider.getType().toString() + ":" + coord.chunkX() + ":" + coord.chunkZ(); return QuantifiedAPI .cache(LC2H.MODID, "lc2h-building-info") .ttl(Duration.ofMinutes(30)) .maxEntries(8192) .diskPreferred() .compressed() .refreshOnAccess() .prefetch(key, () -> buildBuildingInfo(provider, coord)); }
  • Use cache(...) when you care about reusing the actual result value, not just coalescing in-flight work.
  • Keep the cache key tied to real correctness inputs: dimension, chunk, world variant, seed state, whatever can actually change the answer.
  • prefetch(...) is great for warmup or planner systems where you want the value loading now, but you don’t wanna block on some ugly sync path later.
  • V2 auto-registers the mod here too, so the cache bridge no longer needs to front-load manual registration just to function.
Example E - LC²H GPU-first region warmup with honest CPU fallback

LC²H has a region warmup path that tries GPU-assisted batch processing first and falls back to CPU if the backend is unavailable or the drivers are being drivers. That is exactly the kinda job V2 wants expressed as one compute(...) request with backend intent.

Java
RegionBatchProcessingGPUTask workload = new RegionBatchProcessingGPUTask(entries); CompletableFuture<Boolean> future = QuantifiedAPI .<Boolean>compute(LC2H.MODID, "region-processing-batch") .key("region:" + firstRegionKey) .preferVulkan() .fallback(() -> RegionBatchProcessingGPUTask.processBatchOnCPU(entries)) .vulkanWorkload(workload) .openclWorkload(workload) .dataSizeBytes(workload.estimatedVramBytes()) .parallelUnits(workload.estimatedComputeUnits()) .complexity(QuantifiedCompute.Complexity.MASSIVE) .kind(QuantifiedCompute.WorkloadKind.SIMULATION) .timeout(Duration.ofSeconds(45)) .submit();
  • This replaces backend-first public entry points with one intent-first compute request.
  • preferVulkan() says what LC²H wants, while still allowing OpenCL and CPU fallback so the mod doesn’t just die on other hardware.
  • If the workload ain’t actually GPU-worthy, don’t force it. This pattern is best for large spatial batches or genuinely expensive kernel-style math.

How does this actually work?

The architecture is pretty clean once you understand the three-layer model. You only ever touch the top layer while the API does the rest.

Three layers

  • API layer (QuantifiedAPI, request builders) - the surface you code against. Fluent builders that describe what you want.
  • Core runtime (QuantifiedCoreRuntime, async manager, GPU router) - decides how and where work actually runs. CPU thread pool, Vulkan queue, OpenCL context, whatever fits.
  • Platform bridge (loader integration + runtime metadata resolvers) - wires Quantified API into the active loader context and resolves mod metadata across Fabric, Forge, and NeoForge.

Handles and mod identity

On first real use, Quantified API auto-detects the calling mod and creates a QuantifiedHandle for it. All subsequent calls (compute, parallel, cache) go through that handle. It carries your mod ID, version, display name, and the current binding so you don’t have to pass all of that around manually.

If you’re working across threads, using shared helpers between multiple mods, or otherwise crossing context boundaries, use the explicit modId overloads instead of relying on auto-detection. That’s safer and more obvious.

Request builders

ComputeRequest, ParallelRequest, CacheRequest, and QuantifiedTaskGraph.Builder are all fluent builders. They don’t do anything until you call submit(). That means you can construct and pass them around, add conditions, etc., before actually kicking off the work. Pretty handy for conditional GPU paths.

GPU routing

When a ComputeRequest has Vulkan or OpenCL workloads attached, the GPU router (GpuBackendRouter) checks device availability and your GpuBackendPreference. If the preferred backend fails or isn’t available, it falls through to the next option. If everything GPU-side fails, it runs your cpuFallback.

The platform modules live in platforms/forge-1.20.1 and platforms/forge-1.21.1. The core logic that’s platform-agnostic lives in core/. They’re separate Gradle subprojects.

Task lifecycle

What actually happens between .submit() and your future completing? Here’s the full path.

The journey

Build → Normalize → Schedule → Execute → Complete
  • Build: fluent setters collect your key, priority, thread-safety bit, timeout, and optional GPU workloads.
  • Normalize: without GPU workloads, the request becomes a QuantifiedTask for the async manager. With GPU workloads, it becomes a QuantifiedCompute request with backend selection and fallback logic.
  • Schedule: priority maps to an internal PriorityTaskType. AUTO becomes OTHER. foreground() and CRITICAL get the fast lane. background() is low priority and may be deferred.
  • Execute: the async manager picks a thread from the pool, or the GPU router picks a compute queue. Exceptions propagate through the future, not as silent log lines (unless you lose the future reference, but that’s on you xD).
  • Complete: the CompletableFuture resolves. Your thenApply, thenAccept, etc. chains run. If it failed, exceptionally gets called.

Timeouts

You can set a per-task timeout with .timeout(Duration) on a ComputeRequest. There’s also a global default in the config (taskTimeoutMs, defaults to 60s). Per-task overrides win. If a task times out, the future completes exceptionally.

Thread safety bit

ComputeRequest.threadSafe() and notThreadSafe() tell the scheduler whether the work is actually safe to run in parallel. If you mark a task as not thread-safe, the runtime can keep it from overlapping with other work sharing the same affinity lane. Use affinity(String) to group tasks that must not overlap.

For parallel tasks, each item’s mapper function runs independently. The parent future completes when all items are done (or fail, depending on your failure policy). Don’t share mutable state between mapper invocations without synchronization. Yes this sounds obvious, but you’d be surprised ;]

Compute - the workhorse

QuantifiedAPI.compute(taskName) is what you’ll use most xd. It’s the main V2 builder for one unit of work, and it can still route into GPU processing if the task seems fit for it.

Parallelism: Fast multitasking.

Managed concurrency for lists and ranges. This allows you to process items in parallel with smart error handling and strict limits on how many tasks run at once (actually really useful in a lot of cases).

Item-based

Java
CompletableFuture<List<Result>> results = QuantifiedAPI .parallel("processItems") .maxParallelism(4) .failurePolicy(FailurePolicy.FAIL_FAST) // or BEST_EFFORT .items(myList) .map(item -> process(item)) .submit();

Range-based (no list allocation)

Java
QuantifiedAPI.parallel("scanRows") .range(0, height) .map(i -> scanRow(i)) .submit();

Failure policies

  • FAIL_FAST: first exception cancels remaining work and fails the parent future. Use when consistency matters.
  • BEST_EFFORT: keeps going even if some items fail. Errors are collected. Use for batch jobs where partial results are still useful.
During development, use FAIL_FAST so exceptions surface immediately and don’t get swallowed. Switch to BEST_EFFORT only when you deliberately want to tolerate partial failures in production.

ParallelRequest config fields

FieldDefaultNotes
maxParallelismCPU countBound this to avoid overwhelming the pool.
parallelQueueLimit4096Max queued slices. Protects memory if producers go rogue.
parallelMaxSlicesPerMod4096Per-mod slice cap.
failurePolicyFAIL_FASTSee above.

Task graphs - DAG pipelines

When Step B needs the result of Step A, and Step C needs both. That’s a graph. Quantified API’s QuantifiedTaskGraph lets you express dependencies explicitly and runs them in the correct order automatically.

Building a graph

Java
QuantifiedTaskGraph.Builder g = QuantifiedAPI.graph("myPipeline"); NodeHandle<A> a = g.node("load", ctx -> loadData()); NodeHandle<B> b = g.node("process", ctx -> process(ctx.result(a))).dependsOn(a); NodeHandle<C> c = g.node("finalize",ctx -> finalize(ctx.result(b))).dependsOn(b); // Submit and get the terminal result CompletableFuture<C> result = QuantifiedAPI.submitGraph(g, c); // Or get all node results as a Map<String, Object> CompletableFuture<Map<String, Object>> all = QuantifiedAPI.submitGraph(g);

Tips

  • Seed nodes have no dependsOn. They run immediately.
  • Transform nodes use ctx.result(handle) to read a dependency’s output.
  • Multitask / join: call dependsOn with multiple handles on one node to join several branches.
  • Graphs run nodes in topological order. Nodes with no data dependency on each other may run concurrently.
  • If any node throws, the graph future fails. Wire an exceptionally on the result future.
Graphs are great for pipelines where the number of stages is known at construction time. For dynamic multitasking (variable number of items), use parallel instead. Mix them: a graph node can submit a parallel task internally as well.

Keys & identity

Stable keys ARE THE GOLDEN KEYS to efficiency. They make sure the API can reuse work properly instead of getting confused with it. It sounds boring, but it’s actually a huge deal for real-world performance or letting your nod RIP.

What a key does

Every task has a key (a long hash). If two tasks with the same key are submitted before the first completes, Quantified API can deduplicate them and return the same future. This saves work, avoids redundant computation, and helps the caching layer know what to store where.

By default, the key is derived from StableTaskKeys.of("compute", modId, taskName). That’s fine if every submission is different. But if you call the same task with different inputs (e.g. different chunk coords), you need to include those inputs in the key.

Setting a meaningful key

Java
// Chunk-based key: includes dimension + coords .key(dim.location() + ":" + chunkX + "," + chunkZ) // Entity-based key: UUID + dimension is usually enough .key(entity.getStringUUID() + ":" + dim.location()) // Asset key: namespaced resource path for pack reload safety .key(resourceLocation.toString()) // Raw long if you have a stable numeric ID .key((long) chunkX << 32 | (chunkZ & 0xFFFFFFFFL))

Affinity keys

affinity(String) is DIFFERENT. It groups tasks that must not overlap in execution. Use it when your tasks write to a shared mutable state that’s keyed per-dimension or per-region.

Don’t use object references (like chunk.toString() on a mutable object) as keys. If the object state changes, you get different hashes for what should be the same key. Use stable IDs.

Caching

Tiered caching with TTL, size caps, refresh behavior, and optional disk persistence. Mostly when you wanna stop recomputing the same expensive nonsense over and over again.

Basic usage

Java
// Get from cache, compute if missing CompletableFuture<ExpensiveResult> result = QuantifiedAPI .cache("myBucket") .ttl(Duration.ofMinutes(5)) .maxEntries(1024) .memoryOnly() .getAsync(cacheKey, () -> rebuildExpensiveThing()); // Warm it up early without making a weird sync path depend on it QuantifiedAPI.cache("myBucket") .ttl(Duration.ofMinutes(5)) .prefetch(cacheKey, () -> rebuildExpensiveThing()); // Force a background refresh if inputs changed and the old entry is now sus QuantifiedAPI.cache("myBucket") .ttl(Duration.ofMinutes(5)) .refreshAsync(cacheKey, () -> rebuildExpensiveThing()); // Manual eviction on dimension unload QuantifiedAPI.cache("myBucket").remove(keyForDimension); // Nuke the whole bucket QuantifiedAPI.cache("myBucket").clear();

Policy knobs

MethodWhat it does
memoryOnly()Keep the bucket in memory only. Good default for short-lived gameplay state.
diskPreferred()Persist the bucket to disk too. Good for warmup/planner artifacts or expensive derived data.
compressed()Enable compression for persistent caches. Useful when entries are chunky and disk-heavy.
refreshOnAccess()Touch the TTL on reads so hot entries stay alive while they’re actually being used.
fixedTtl()Do not extend TTL on reads. Better when you want old entries to die on schedule no matter what.

Load helpers

MethodBest for
get(key, loader)Simple sync lookup with compute-on-miss.
getAsync(key, loader)Async lookup with compute-on-miss.
prefetch(key, loader)Warmup or preloading before the value is urgently needed.
refresh(key, loader)Force a synchronous replacement of an existing entry.
refreshAsync(key, loader)Force a background replacement of an entry when the old value is stale or invalid.

Practical advice

  • Set maxEntries. Without it, caches can grow stupidly large and eat your heap.
  • Use memoryOnly() for fast ephemeral state and diskPreferred() for expensive artifacts worth surviving a restart.
  • Use refreshOnAccess() when the same hot entries get hammered a lot. Use fixedTtl() when you want time-based expiry to stay strict.
  • Use prefetch(...) for warmup systems. Use refreshAsync(...) when the old value exists but is now out of date.
  • Clear buckets on dimension unload events. Stale per-dimension data after unload is a memory leak.
  • Use separate buckets for logically separate data. Don’t throw chunk data, entity data, and planner artifacts into one blob bucket.
Don’t confuse compute(...).key(...) with cache(...). The compute key is for in-flight identity and dedupe. The cache bucket is for stored value reuse. You will often want both.

GPU routing

Vulkan is the preffered one, OpenCL is the fallback. Quantified API at runtime figures out what’s available and routes accordingly. You can also tell the API what you want :]

How the router works

When you call submit() on a ComputeRequest with GPU workloads, the GpuBackendRouter checks:

  1. Checks what you asked for: preferVulkan(), preferOpenCL(), requireVulkan(), requireOpenCL(), preferGpu(), or cpuOnly().
  2. If the preferred backend is available and your workload is ready/attached for it, Quantified API tries that backend first.
  3. If the preferred GPU path misses/bonks and the request was only a preference, it can still try the other GPU backend if your code can use it. :]
  4. If both GPU backends fail or are unavailable, it falls back to your fallback(...) CPU path (DON’T FORGET TO INCLUDE A CPU FALLBACK!!!)
  5. If no fallback is provided and the GPU fails, the future completes exceptionally.
Always provide a fallback(...) when using GPU workloads. If the player’s GPU doesn’t support your shader, or Vulkan probe fails, or drivers are weird (and drivers are ALWAYS weird), or the player is running the API on a server, you need somewhere to send the compute tasks. No fallback = everything goes boom :[

Backend preferences

PreferenceBehavior
preferGpu()Lets Quantified API pick the best GPU lane it can justify, while still keeping CPU fallback open.
preferVulkan()Biases routing toward Vulkan first, then other valid fallbacks.
preferOpenCL()Biases routing toward OpenCL first, which can be nicer for wider hardware compat.
requireVulkan()Only Vulkan is acceptable for the GPU path. If that cannot run, QAPI drops to CPU fallback instead of casually swapping GPU backends.
requireOpenCL()Same idea, but for OpenCL.
cpuOnly()Skips the GPU entirely. Handy for debugging, tiny workloads, or server-side-only paths.

Vulkan probe

Quantified API runs a Vulkan probe at startup to check device capabilities. Probe results live in quantified/embedded/vulkanProbe/ inside the mod resources. The actual shaders (GLSL compute, compiled to SPIR-V) are in quantified/shaders/src/vulkan/.

Errors & failure policies

Don’t expect the API to quietly bury your mistakes (like its fine.... it aint). Unhandled errors spread fast and break everything they touch (like cancer... its bad). Here is how to catch them properly so you can keep your code (and your sanity) intact (hopefully...).

The golden rule

Always keep a reference to your futures and always attach an exceptionally or handle somewhere in the chain. If you fire-and-forget a future and it fails silently, that’s on you. Quantified API surfaces the error, you just need to see it.

Java
QuantifiedAPI.compute("riskyWork") .submit(() -> doThing()) .thenAccept(result -> useResult(result)) .exceptionally(ex -> { LOGGER.error("riskyWork failed: {}", ex.getMessage()); return null; });

Parallel failure policies

  • FAIL_FAST: first exception cancels remaining work. Future fails. Great for development, good for consistent batch operations.
  • BEST_EFFORT: continues even if some items fail. Use for batch jobs where partial results are acceptable. Check the result carefully.
With BEST_EFFORT on parallel tasks, individual item failures may be swallowed if you don’t inspect the combined result object. Switch to FAIL_FAST during development to catch exceptions early.

IllegalStateException

In V2, this is no longer the normal failure mode. compute, parallel, graph, and cache will usually auto-register your mod on first use. If you still hit an IllegalStateException, it usually means auto-detection could not tell which mod owns the call because you are inside shared helper code, a test harness, an unusual bootstrap path, or some other ambiguous context. The fix is to use the explicit modId overloads or call register(modId) as a manual override.

Troubleshooting

Look, we both know why you're here xd. Something broke, isn't performing well or you're experiencing the peak of modding - the infamous red and white logs (R.I.P). It happens to all of us, and these tips should help ya out :]

QAPI couldn’t auto-detect my mod :[

Auto-detection usually works, but helper classes, test environments, static initializers, shared libs, or cross-mod utilities can make ownership ambiguous. Fix: use the explicit modId overloads like compute(modId, task), or call register(modId) as a manual override. Really that’s it. xD

TPS drops after adding GPU workloads

GPU path might be synchronizing too eagerly or stalling on buffer uploads. Check the dashboard GPU pressure metrics. Also verify you’re not accidentally calling blocking OpenGL/LWJGL operations from async threads (that’ll go badly). Reduce gpuMaxConcurrent if the GPU is overwhelmed. Check fallback paths with CPU_ONLY mode to confirm the issue is GPU-specific.

Parallel map seems to silently “lose” errors

You’re using BEST_EFFORT and not inspecting the result for partial failures. Switch to FAIL_FAST during development. In production with BEST_EFFORT, check the combined result object for per-item errors.

Caches growing without bound

You forgot maxEntries or your TTL is way too long. Also check that you’re calling cache.clear() or cache.remove(key) on dimension unload events. Per-dimension data that survives dimension unload is a memory leak.

Future never completes

Check if your supplier or mapper is blocking indefinitely (e.g. waiting for a lock that’s held by the main thread, which is waiting for YOUR future). Classic deadlock. Use .timeout(Duration) to surface this instead of hanging. Enable debugShowTimings to see which tasks are stuck.

When filing a bug: grab a dashboard export, the relevant section of logs/latest.log, your Quantified API version string, and your config file. For GPU issues, also grab the GPU diagnostics section from the dashboard. Stack trace alone is usually not enough context ;]
