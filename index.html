Hey, welcome to Quantified API! :D

So you're building a Minecraft mod and your async code is doing that fun thing where it works 90% of the time, and the other 10% it just... vibes incorrectly. Threads fighting, futures vanishing, tick thread crying. xD

Quantified API is a runtime layer that handles scheduling, threading, GPU compute, caching, and observability. You write the logic. Quantified API figures out where and when to run it. ;]

How it actually works

The internals are interesting! Handles, routers, async managers. Worth a read even if you don’t need it immediately.

Okay but why does this exist

Mods love to bolt CompletableFuture.runAsync directly onto vanilla code paths. That scatters work across unnamed thread pools, competes with the server tick, duplicates the same boilerplate across every module, and eventually explodes when someone blocks on the main thread waiting for a future that resolves slightly later than expected. And then the TPS graph goes brrrr downwards. xD

Quantified API gives you one central place for identity (per-mod handles so work is always attributed), stable keys (deduplication and cache locality), routing (CPU vs GPU with honest fallbacks), and observability (dashboard + telemetry so you can actually see what’s happening). You still write the math. Quantified API owns the plumbing.

Call QuantifiedAPI.register(modId, displayName, version) from your mod constructor or a dedicated setup method, not lazily on first use. If the handle is missing when you call compute, parallel, or cache, you get an IllegalStateException. Just register early and forget about it :)

Pick the right tool for the job

What you needWhat to useWhy
One async result QuantifiedAPI.compute(name) Fluent builder. Supports GPU workloads with CPU fallback.
Many items in parallel parallel(...).items(...).map(...) Controlled fan-out, configurable failure policies.
Index-based loops parallel(...).range(a, b).map(...) Same engine, no intermediate list allocation.
Complex pipelines with deps QuantifiedAPI.graph(modId, name) DAG execution, context.result(handle) for wiring.
Cache expensive pure work cache("bucket").get(key, loader) TTL, size caps, optional disk persistence.
Fire-and-forget side effect QuantifiedAPI.runAsync(name, () -> ...) Thin wrapper over compute, no return value.
Prefer composing CompletableFuture with thenApply and exceptionally instead of calling .join() on the server thread. Blocking there shows up as lag. If you genuinely need to block in a safe window (world load, data fixer), comment it in code review so the next person knows it’s intentional.

Let’s get you up and running :DD

Three steps. Actually more like two and a half. You’ll be submitting async jobs within 5 minutes, I promise xD

Step 1 - Tell Quantified API your mod exists

Call register from your mod constructor or early setup. Do it once. Done.

Java
// In your mod constructor or setup event public class MyMod { public MyMod() { // Simple: pulls name + version from mods.toml automatically QuantifiedAPI.register("mymod"); // Or be explicit: QuantifiedAPI.register("mymod", "My Cool Mod", "1.0.0"); } }
Skip registration and your first compute() or parallel() call will throw IllegalStateException: QuantifiedAPI not initialized. Which is annoying. So just register first. :)

Step 2 - Submit some work

CPU compute is the most common case. You get back a CompletableFuture. Chain on it, don’t block on it from the tick thread.

Java - async compute
CompletableFuture<Digest> future = QuantifiedAPI .compute("chunkDigest") // task name (for metrics + keys) .key(dimId + "|" + cx + "|" + cz) // stable dedup key .foreground() // or .background() for low priority .submit(() -> expensiveHash(snapshot)); future.thenAccept(digest -> { // runs off the tick thread - safe! storeResult(digest); });

Step 3 - Parallel over a list (the fun part xD)

Java - parallel map
CompletableFuture<List<VoxelMesh>> meshes = QuantifiedAPI .parallel("bakeMeshes") .maxParallelism(Math.min(8, Runtime.getRuntime().availableProcessors())) .items(partitions) .map(p -> bake(p)) .submit(); // Range-based (no list needed): QuantifiedAPI.mapRange("processRows", 0, height, i -> processRow(i));

Handy shorthands

  • QuantifiedAPI.forEach(name, collection, consumer) - parallel side effects, returns when all done.
  • QuantifiedAPI.mapOrdered(name, collection, mapper) - parallel but preserves list order.
  • QuantifiedAPI.mapRange(name, start, end, fn) - index-based parallel, no stream boilerplate.
  • QuantifiedAPI.runAsync(name, runnable) - fire and forget, no result.
If your mod separates bootstrap from execution contexts (common in multi-module setups), use the explicit modId overloads: compute(modId, task), parallel(modId, task), graph(modId, name). That bypasses the ThreadLocal handle lookup and avoids confusion when helper classes are shared.

Before you ship, check these

  • Every hot-path task has a stable key when the same task name is reused across different inputs.
  • GPU code paths have a CPU fallback that produces equivalent output.
  • Your parallel failure policy matches what you actually want (FAIL_FAST vs BEST_EFFORT).
  • You’ve opened developerDashboard: true at least once and poked around. It’s cool ;]

Copy-paste recipes :D

Real patterns from real use cases. Grab what fits, tweak the names, ship it. These are tested shapes, not theoretical ones.

Recipe A - Worldgen chunk prefetch

Precompute expensive per-chunk data off the tick thread. Cache the result so repeat access is free.

Java
public CompletableFuture<ChunkData> prefetch(ChunkPos pos, ResourceKey<Level> dim) { String key = dim.location() + ":" + pos.x + "," + pos.z; return QuantifiedAPI.cache("chunkData") .ttl(Duration.ofMinutes(10)) .maxEntries(4096) .get(key, () -> QuantifiedAPI .compute("chunkAnalysis") .key(key) .background() .submit(() -> analyzeChunk(pos, dim)) ); }
Recipe B - GPU kernel with honest CPU fallback

Attempt GPU, fall back to CPU automatically. Quantified API picks the backend based on your hints and what the hardware supports.

Java
CompletableFuture<float[]> result = QuantifiedAPI .compute("terrainNoise") .vulkanWorkload(ctx -> ctx.runShader("terrain.comp", inputBuffer)) .openclWorkload(ctx -> ctx.runKernel("terrain.cl", inputBuffer)) .cpuFallback(() -> computeOnCpu(inputBuffer)) // always provide this! .complexity(QuantifiedCompute.Complexity.HEAVY) .kind(QuantifiedCompute.WorkloadKind.SIMULATION) .backendPreference(GpuBackendPreference.VULKAN_FIRST) .submit(null); // null = no extra CPU supplier, GPU paths + fallback cover it
Recipe C - DAG pipeline (seed → transform → reduce)

When Step B needs the output of Step A, use a graph. Nodes declare dependencies explicitly.

Java
QuantifiedTaskGraph.Builder g = QuantifiedAPI.graph("lightingPipeline"); // Step A: load raw data QuantifiedTaskGraph.NodeHandle<RawData> load = g.node("load", ctx -> loadRawData()); // Step B: depends on A QuantifiedTaskGraph.NodeHandle<ProcessedData> process = g.node("process", ctx -> process(ctx.result(load))) .dependsOn(load); // Step C: depends on B, terminal result QuantifiedTaskGraph.NodeHandle<LightMap> reduce = g.node("reduce", ctx -> reduce(ctx.result(process))) .dependsOn(process); CompletableFuture<LightMap> future = QuantifiedAPI.submitGraph(g, reduce);
Recipe D - Debounced background cache refresh

Regenerate expensive computed data in the background whenever inputs change, without hammering the thread pool.

Java
private volatile CompletableFuture<SomeResult> pendingRefresh; public void onInputChanged() { if (pendingRefresh != null && !pendingRefresh.isDone()) { return; // already refreshing, chill xD } pendingRefresh = QuantifiedAPI .compute("bgRefresh") .background() .submit(this::rebuildData) .thenAccept(result -> SomeCache.update(result)); }

How this actually works ;]

The architecture is pretty clean once you understand the three-layer model. You only ever touch the top layer. The rest happens for you.

Three layers

  • API layer (QuantifiedAPI, request builders) - the surface you code against. Fluent builders that describe what you want.
  • Core runtime (QuantifiedCoreRuntime, async manager, GPU router) - decides how and where work actually runs. CPU thread pool, Vulkan queue, OpenCL context, whatever fits.
  • Platform bridge (Forge 1.20.1 / 1.21.1 modules) - wires Quantified API into the Forge event lifecycle, mods.toml metadata, and server/client contexts.

Handles and mod identity

When you call register, Quantified API creates a QuantifiedHandle for your mod ID. All subsequent calls (compute, parallel, cache) go through that handle. It carries your mod ID, version, and a ThreadLocal reference so you don’t have to pass it everywhere.

If you’re working across threads or sharing helpers between mods, use the explicit modId overloads instead of relying on the thread-local. That’s just safer and more obvious.

Request builders

ComputeRequest, ParallelRequest, CacheRequest, and QuantifiedTaskGraph.Builder are all fluent builders. They don’t do anything until you call submit(). That means you can construct and pass them around, add conditions, etc., before actually kicking off the work. Pretty handy for conditional GPU paths.

GPU routing

When a ComputeRequest has Vulkan or OpenCL workloads attached, the GPU router (GpuBackendRouter) checks device availability and your GpuBackendPreference. If the preferred backend fails or isn’t available, it falls through to the next option. If everything GPU-side fails, it runs your cpuFallback. Always provide one!

The platform modules live in platforms/forge-1.20.1 and platforms/forge-1.21.1. The core logic that’s platform-agnostic lives in core/. They’re separate Gradle subprojects.

Task lifecycle

What actually happens between .submit() and your future completing? Here’s the full path.

The journey

Build → Normalize → Schedule → Execute → Complete
  • Build: fluent setters collect your key, priority, thread-safety bit, timeout, and optional GPU workloads.
  • Normalize: without GPU workloads, the request becomes a QuantifiedTask for the async manager. With GPU workloads, it becomes a QuantifiedCompute request with backend selection and fallback logic.
  • Schedule: priority maps to an internal PriorityTaskType. AUTO becomes OTHER. foreground() and CRITICAL get the fast lane. background() is low priority and may be deferred.
  • Execute: the async manager picks a thread from the pool, or the GPU router picks a compute queue. Exceptions propagate through the future, not as silent log lines (unless you lose the future reference, but that’s on you xD).
  • Complete: the CompletableFuture resolves. Your thenApply, thenAccept, etc. chains run. If it failed, exceptionally gets called.

Timeouts

You can set a per-task timeout with .timeout(Duration) on a ComputeRequest. There’s also a global default in the config (taskTimeoutMs, defaults to 60s). Per-task overrides win. If a task times out, the future completes exceptionally.

Thread safety bit

ComputeRequest.threadSafe(boolean) is a hint to the scheduler. Default is true. If you mark a task as not thread-safe, the scheduler may serialize it with other tasks sharing the same affinity key. Use affinityKey(String) to group tasks that must not overlap.

For parallel tasks, each item’s mapper function runs independently. The parent future completes when all items are done (or fail, depending on your failure policy). Don’t share mutable state between mapper invocations without synchronization. Yes this sounds obvious, but you’d be surprised ;]

Compute - the workhorse

QuantifiedAPI.compute(taskName) is what you’ll use most. It’s a fluent builder for a single async task that can optionally run on GPU.

Key ComputeRequest methods

MethodWhat it does
.key(String)Stable dedup key. Tasks with the same key get the same hash-based ID. Use dimension + coords for world stuff.
.foreground()High priority. For work the player will notice if it’s late.
.background()Low priority. For prewarming, analytics, anything that can wait.
.priority(ExecutionPriority)Full control: AUTO, BACKGROUND, FOREGROUND, CRITICAL.
.timeout(Duration)Per-task timeout. Overrides global config default.
.threadSafe(boolean)Hint to scheduler. Default true.
.affinityKey(String)Group tasks that must not run concurrently.
.vulkanWorkload(fn)Vulkan compute path. Provide a cpuFallback too!
.openclWorkload(fn)OpenCL path. Used as fallback to Vulkan when VULKAN_FIRST.
.cpuFallback(Supplier)Always-safe path. Required if you have GPU workloads.
.complexity(Complexity)SIMPLE, MODERATE, HEAVY, MASSIVE - affects cost model and GPU queue selection.

Parallel - fan-out, done right

When you have a list (or a range) and you want to process every item concurrently with sane failure handling and bounded concurrency. This is the thing.

Item-based

Java
CompletableFuture<List<Result>> results = QuantifiedAPI .parallel("processItems") .maxParallelism(4) .failurePolicy(FailurePolicy.FAIL_FAST) // or BEST_EFFORT .items(myList) .map(item -> process(item)) .submit();

Range-based (no list allocation)

Java
QuantifiedAPI.parallel("scanRows") .range(0, height) .map(i -> scanRow(i)) .submit();

Failure policies

  • FAIL_FAST: first exception cancels remaining work and fails the parent future. Use when consistency matters.
  • BEST_EFFORT: keeps going even if some items fail. Errors are collected. Use for batch jobs where partial results are still useful.
During development, use FAIL_FAST so exceptions surface immediately and don’t get swallowed. Switch to BEST_EFFORT only when you deliberately want to tolerate partial failures in production.

ParallelRequest config fields

FieldDefaultNotes
maxParallelismCPU countBound this to avoid overwhelming the pool.
parallelQueueLimit4096Max queued slices. Protects memory if producers go rogue.
parallelMaxSlicesPerMod4096Per-mod slice cap.
failurePolicyFAIL_FASTSee above.

Task graphs - DAG pipelines

When Step B needs the result of Step A, and Step C needs both. That’s a graph. Quantified API’s QuantifiedTaskGraph lets you express dependencies explicitly and runs them in the correct order automatically.

Building a graph

Java
QuantifiedTaskGraph.Builder g = QuantifiedAPI.graph("myPipeline"); NodeHandle<A> a = g.node("load", ctx -> loadData()); NodeHandle<B> b = g.node("process", ctx -> process(ctx.result(a))).dependsOn(a); NodeHandle<C> c = g.node("finalize",ctx -> finalize(ctx.result(b))).dependsOn(b); // Submit and get the terminal result CompletableFuture<C> result = QuantifiedAPI.submitGraph(g, c); // Or get all node results as a Map<String, Object> CompletableFuture<Map<String, Object>> all = QuantifiedAPI.submitGraph(g);

Tips

  • Seed nodes have no dependsOn. They run immediately.
  • Transform nodes use ctx.result(handle) to read a dependency’s output.
  • Fan-out / join: call dependsOn with multiple handles on one node to join several branches.
  • Graphs run nodes in topological order. Nodes with no data dependency on each other may run concurrently.
  • If any node throws, the graph future fails. Wire an exceptionally on the result future.
Graphs are great for pipelines where the number of stages is known at construction time. For dynamic fan-out (variable number of items), use parallel instead. Mix them: a graph node can submit a parallel task internally. xD

Keys & identity

Stable keys are how Quantified API deduplicates work and keeps cache locality sane. They sound boring but they matter a lot in practice. Trust me on this one :)

What a key does

Every task has a key (a long hash). If two tasks with the same key are submitted before the first completes, Quantified API can deduplicate them and return the same future. This saves work, avoids redundant computation, and helps the caching layer know what to store where.

By default, the key is derived from StableTaskKeys.of("compute", modId, taskName). That’s fine if every submission is different. But if you call the same task with different inputs (e.g. different chunk coords), you need to include those inputs in the key.

Setting a meaningful key

Java
// Chunk-based key: includes dimension + coords .key(dim.location() + ":" + chunkX + "," + chunkZ) // Entity-based key: UUID + dimension is usually enough .key(entity.getStringUUID() + ":" + dim.location()) // Asset key: namespaced resource path for pack reload safety .key(resourceLocation.toString()) // Raw long if you have a stable numeric ID .key((long) chunkX << 32 | (chunkZ & 0xFFFFFFFFL))

Affinty keys

affinityKey(String) is different. It groups tasks that must not overlap in execution. Use it when your tasks write to shared mutable state that’s keyed per-dimension or per-region.

Don’t use object references (like chunk.toString() on a mutable object) as keys. If the object state changes, you get different hashes for what should be the same key. Use stable IDs.

Caching

Tiered caching with TTL, size caps, and optional disk persistence. For when you compute something expensive once and want to reuse it without thinking about it.

Basic usage

Java
// Get from cache, compute if missing CompletableFuture<ExpensiveResult> result = QuantifiedAPI .cache("myBucket") .ttl(Duration.ofMinutes(5)) .maxEntries(1024) .get(cacheKey, () -> QuantifiedAPI.compute("recompute").submit(loader)); // Manual eviction on dimension unload QuantifiedAPI.cache("myBucket").remove(keyForDimension); // Nuke the whole bucket QuantifiedAPI.cache("myBucket").clear();

Practical advice

  • Set maxEntries. Without it, caches can grow unbounded and eat your heap. That’s bad.
  • Pick TTLs tied to gameplay time, not wall time. “5 minutes” of server time is more meaningful than 5 real minutes.
  • Clear buckets on dimension unload events. Stale per-dimension data after unload is a memory leak.
  • Use separate buckets for logically separate data. Don’t throw chunk data and entity data in the same bucket.
The cache loader is a Supplier<CompletableFuture<T>>. That means the loader itself can submit async work. Cache + compute stack cleanly together for prefetch patterns.

GPU routing

Vulkan-first, OpenCL as fallback, honest CPU escape hatch. Quantified API figures out what’s available at runtime and routes accordingly. You just describe what you want.

How the router works

When you call submit() on a ComputeRequest with GPU workloads, the GpuBackendRouter checks:

  1. Is Vulkan available and initialized? If yes and preference is VULKAN_FIRST or AUTO, try Vulkan.
  2. If Vulkan fails or is unavailable, try OpenCL if you provided an OpenCL workload.
  3. If both GPU paths fail or are unavailable, fall back to your cpuFallback.
  4. If no fallback is provided and GPU fails, the future completes exceptionally.
Always provide a cpuFallback when using GPU workloads. If the player’s GPU doesn’t support your shader, or Vulkan probe fails, or drivers are weird (and drivers are ALWAYS weird), you need somewhere to land. No fallback = mod breaks on some hardware. :((

Backend preferences

PreferenceBehavior
AUTOQuantified API decides based on device capabilities and workload hints.
VULKAN_FIRSTTry Vulkan, then OpenCL, then CPU.
OPENCL_FIRSTTry OpenCL first, then CPU. Useful for wider compat.
CPU_ONLYSkip GPU entirely. Handy for debugging or server-side-only paths.

Vulkan probe

Quantified API runs a Vulkan probe at startup to check device capabilities. Probe results live in quantified/embedded/vulkanProbe/ inside the mod resources. The actual shaders (GLSL compute, compiled to SPIR-V) are in quantified/shaders/src/vulkan/.

Errors & failure policies

Exceptions don’t get swallowed in Quantified API. They propagate through the future chain. Here’s how to handle them properly without crying.

The golden rule

Always keep a reference to your futures and always attach an exceptionally or handle somewhere in the chain. If you fire-and-forget a future and it fails silently, that’s on you. Quantified API surfaces the error, you just need to listen.

Java
QuantifiedAPI.compute("riskyWork") .submit(() -> doThing()) .thenAccept(result -> useResult(result)) .exceptionally(ex -> { LOGGER.error("riskyWork failed: {}", ex.getMessage()); return null; });

Parallel failure policies

  • FAIL_FAST: first exception cancels remaining work. Future fails. Great for development, good for consistent batch operations.
  • BEST_EFFORT: continues even if some items fail. Use for batch jobs where partial results are acceptable. Check the result carefully.
With BEST_EFFORT on parallel tasks, individual item failures may be swallowed if you don’t inspect the combined result object. Switch to FAIL_FAST during development to catch exceptions early.

IllegalStateException

You get this if you call compute, parallel, or cache before register. The fix is always the same: register earlier. Put it in your mod constructor. It takes one line. xD

Monitoring & telemetry

Quantified API ships with a built-in webpanel and metrics system. Turn it on, poke around, and actually see what your async code is doing. It’s really useful, I promise :D

Developer dashboard

Set "developerDashboard": true in the config to enable it. The dashboard runs as a local HTTP server (port configurable, default something sensible). Open it in a browser while the game is running.

You’ll see: task throughput, queued work, GPU metrics (if applicable), per-mod stats, error rates, and timing histograms. It’s actually kind of cool to watch in real time while stress testing.

Troubleshooting

The issues that actually come up. Quick answers, no fluff.

“QuantifiedAPI not initialized”

You called compute/parallel/cache before register, OR you lost the ThreadLocal context (common in cross-mod helper classes or static initializers). Fix: register as early as possible (mod constructor), and use the explicit modId overloads in helpers. Really that’s it. xD

TPS drops after adding GPU workloads

GPU path might be synchronizing too eagerly or stalling on buffer uploads. Check the dashboard GPU pressure metrics. Also verify you’re not accidentally calling blocking OpenGL/LWJGL operations from async threads (that’ll go badly). Reduce gpuMaxConcurrent if the GPU is overwhelmed. Check fallback paths with CPU_ONLY mode to confirm the issue is GPU-specific.

Parallel map seems to silently “lose” errors

You’re using BEST_EFFORT and not inspecting the result for partial failures. Switch to FAIL_FAST during development. In production with BEST_EFFORT, check the combined result object for per-item errors.

Caches growing without bound

You forgot maxEntries or your TTL is way too long. Also check that you’re calling cache.clear() or cache.remove(key) on dimension unload events. Per-dimension data that survives dimension unload is a memory leak.

Future never completes

Check if your supplier or mapper is blocking indefinitely (e.g. waiting for a lock that’s held by the main thread, which is waiting for YOUR future). Classic deadlock. Use .timeout(Duration) to surface this instead of hanging. Enable debugShowTimings to see which tasks are stuck.

When filing a bug: grab a dashboard export, the relevant section of logs/latest.log, your Quantified API version string, and your config file. For GPU issues, also grab the GPU diagnostics section from the dashboard. Stack trace alone is usually not enough context. ;]

Publishing your mod

If Quantified API is on your classpath as a real dependency - not just mentioned in readme - BRSSLA v1.5 treats your mod alone in a deliberately open way when it comes to commercial distribution.

In plain terms: when you ship a mod that properly depends on Quantified API, you generally do not need a separate commercial-license deal with BlackRift Studios just to sell your mod or offer paid access to builds. Commercial rights attach to that dependent mod project so teams can monetize work they built on top of the API without negotiating a bespoke contract for vanilla Quantified API availability.

The scope stays narrow on purpose. It does not hand you blanket rights over Quantified API itself: you still cannot relicense the library wholesale, bundle its source outside what the agreement allows, or stretch this carve-out onto every downstream fork or unrelated product. Grants that apply specifically to dependents may also be spelled out further in the repository readme - read those lines before cutting a trailer or storefront page.
